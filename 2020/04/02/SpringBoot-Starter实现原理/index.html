自定义SpringBoot-Starter

springboot中，最常用的就是一种“可插拔”式的依赖就是starter，比如常见的spring-boot-starter-jdbc，spring-boot-starter-data-mongodb。

starter相当于一个非侵入的，具备完整功能性的，且遵循（依赖）SpringBoot设计概念的一种组件。

自己写一个简单的Starter

首先，为了尽可能的仿造一个starter，设计一个接口：

public interface Car {
    void drive();
}

然后，他的两个实现类，宝马车：

public class BCar implements Car {
    public void drive() {
        System.out.println("BWM");
    }
}

特斯拉车：

public class TCar implements Car {
    public void drive() {
        System.out.println("Tesla");
    }
}

把实现类都注册为Bean：

@Configuration
public class CarConfig {

    @Bean
    @Primary
    @ConditionalOnMissingClass("com.alibaba.fastjson.JSON")
    public BCar bCar(){
        return new BCar();
    }

    @Bean
    @ConditionalOnClass(name = "com.alibaba.fastjson.JSON")
    public TCar tCar(){
        return new TCar();
    }

}

以上这几个步骤都很基础，接下来是实现starter的核心了：

@Configuration
//引入CarConfig组件（也就是加载Bean）
@Import(CarConfig.class)
//开启CarProperties属性的自动配置
@EnableConfigurationProperties(CarProperties.class)
public class Register {

    @Bean
    public CarTemplate carTemplate(CarProperties carProperties,Car car) {
        Map<String, Object> info = carProperties.getInfo();
        System.out.println("Map中含有" + info);
        return new CarTemplate(carProperties,car);
    }

}

这里之所以要配置properties属性，就是为了尽可能的仿造。因为像mybatis,redis这种的starter，不管它到底是什么东西，它们为了要完成某些功能就一定要配置相应的属性，比如数据库名、密码、连接地址…所以我们的starter并不是为了写而写（当然，现在是这样的），starter是为了完成某一项功能而开发，它同样需要配置它所需要的属性：

//所有的，配置文件中前缀为指定值的配置，都要加载进来
@ConfigurationProperties(prefix = CarProperties.CAR)
public class CarProperties {
    public static final String CAR = "guo.car";

    //所有的，前缀为guo.car.info的属性
    //比如：guo.car.info.name=GuoTianChi
    //Map特殊，如果是String，就必须为guo.car.info=GuoTianChi
    private Map<String,Object> info;

    //所有的，前缀为guo.car.is-check的属性
    //注意，必须要有set和get方法；是“和”，两者都要有
    //有了set/get方法才可以赋值/取值，properties的配置才有意义
    private boolean isCheck;

    public Map<String, Object> getInfo() {
        return info;
    }

    public void setInfo(Map<String, Object> info) {
        this.info = info;
    }

    public void setCheck(boolean check) {
        isCheck = check;
    }

    public boolean isCheck() {
        return isCheck;
    }

    //内部类的定义要特殊，没有了set，只有get（必须有）
    //这里的内部类就是为了更好的区分各个子块的关系
    //配置文件就可以配置为：guo.car.inner.should=true
    private final Inner inner = new Inner();

    public Inner getInner() {
        return inner;
    }
    public static class Inner {
        private boolean isShould;

        public boolean isShould() {
            return isShould;
        }

        public void setShould(boolean should) {
            isShould = should;
        }

}

其实，有了上面几个类就可以满足我们的需求了，在需要时，用maven引入这个starter的jar包，然后怎么使用呢：

//如果想用宝马：
Car carB = new BCar();
carB.drive();
//想用特斯拉：
Car carT = new TCar();
carT.drive();

十分的不优雅，这也就丧失了springboot-starter的意义。

所以，我们的目标是，通过某些属性，动态的、通过spring注入Car。比如，当项目中有了German类，就加载宝马，没有的话就加载特斯拉。怎么实现呢？在这里：

//通过fastjson代替一下German
@ConditionalOnMissingClass("com.alibaba.fastjson.JSON")

然后，怎么利用springboot的自动配置，顺便帮我们的这个starter直接给配置了多好，在resources—>META-INF—>spring.factories：

org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
  com.sky.guo.myself.starter.Register

最后，怎么注入呢，想到了一种模板设计模式：

public class CarTemplate {

    private CarProperties carProperties;
    private Car car;

    public CarTemplate(CarProperties carProperties, Car car) {
        this.carProperties = carProperties;
        this.car = car;
    }

    public void execute(){
        car.drive();
        System.out.println(this.carProperties);
    }

}

所以，截至到现在，整体的逻辑是：springboot在启动时，正常运行它的自动配置，顺便扫到了我们资源路径下的spring.factories，发现其内容符合自动配置，就帮助我们加载了com.sky.guo.myself.starter.Register，而Register中做了几件事：

  1. 引入了CarConfig，所以也就是注册了Car的Bean；
  2. 开启CarProperties属性的自动配置，使得用户可以配置对应属性
